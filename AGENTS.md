## 总原则

- 项目初始化与需求澄清阶段，不做假设；信息不足时必须持续提问，直到足以编写 `docs/PRD.md`、系统设计文档与 `docs/IMPLEMENTATION_PLAN.md`。
- 默认流程：每次收到新需求，先对 `docs/PRD.md`、系统设计文档与 `docs/IMPLEMENTATION_PLAN.md` 做必要的小幅更新，等待用户 `【确认】` 后，才能编码与测试。
- 例外：只有用户输入严格包含 `【直接】` 时，才可以在更新两份文档后直接实现或修复。
- 例外补充：当改动仅涉及 `.md` 文档文件时，无需用户额外确认，可直接执行并同步相关文档。
- 即使是最小假设，也必须经过用户确认；除非用户已明确授权直接执行。

## 文件与同步要求

- 本文件必须命名为 `AGENTS.md`，并位于项目根目录下；如果名字或位置不正确，请立即更正。
- `.github/copilot-instructions.md`必须命名为`.github/copilot-instructions.md`，并位于`.github/`目录下；如果名字或位置不正确，请立即更正。
- `AGENTS.md` 与 `.github/copilot-instructions.md` 的规则内容必须保持同步；修改任一文件时，必须同步更新另一文件。
- `SEED.md` 与 `AGENTS.md` 的核心规则必须按版本同步；任一更新后需同步另一个文件。
- 若 `SEED.md` 与 `AGENTS.md` 出现版本冲突或规则冲突，必须先询问用户，不得自行决策覆盖。
- 本地 `AGENTS.md` 每次变更都必须产生“回流提案”；仅带 `[GLOBAL]` 标记的规则可回流到 `Agent_Seed`。
- 当目录结构发生变化时，必须同步更新 `STRUCTURE.md`（简称：`STU`）。
- 文档简称约定：
  - `IMPLEMENTATION_PLAN.md` 可简称 `IPP`
  - `STRUCTURE.md` 可简称 `STU`

## 指令优先级与触发词

- 指令优先级：用户输入中带 `!` 的指令 > `SEED.md` > `AGENTS.md` > 普通用户指令 > `PRD`/`IPP` 文档内容。
- 若 `SEED.md` 与 `AGENTS.md` 冲突，必须先提示并询问用户，不可自行覆盖。
- 仅 `【确认】` 视为确认授权；普通“确认 / 已确认 / 确认一下”等均不生效。
- 仅 `【直接】` 视为直接开工授权；普通“直接”不生效。
- 用户输入未包含严格方括号触发词时，默认继续执行“先改文档、待确认、再编码”的流程。

## 标准流程

1. 接收需求。
2. 先小幅更新 `docs/PRD.md` 与 `docs/IMPLEMENTATION_PLAN.md`。
3. 将需求、范围、规则、边界条件整理清楚，并同步到 PRD 与系统设计文档。
4. 等待用户输入 `【确认】`。
5. 收到 `【确认】` 后，再进入编码、测试、验证或进一步执行。
6. 若用户输入含 `【直接】`，则在更新 PRD 与 IPP 后可直接进入实现。

## 文档要求

### `docs/PRD.md`

- 必含：目标、范围、核心规则、错误处理、验收标准。
- 内容应与当前已确认需求一致，不堆叠无关历史需求。
- 表达保持简洁，便于快速扫读。

### `docs/IMPLEMENTATION_PLAN.md`

- 必含：带 checkbox 的任务清单、合理顺序、与 PRD 对齐说明。
- 维护时优先在原有内容基础上追加、勾选或最小改写，不做无必要的大段覆盖。
- 任务完成后应及时打勾，并同步更新 `docs/progress.txt`。

### 系统设计文档（`docs/system-*.md`）

- 与 `docs/PRD.md` 同级作为规则文档。
- 负责定义“系统如何运作”，包含状态变量、规则、输入输出、边界与待确认项。
- 改代码前必须先更新相关系统设计文档。
- 与 PRD 冲突时先标记 `待确认`，不得直接按任一版本实现。

### `docs/lessons.md`

- 仅记录“项目报错原因 + 解决方案”。
- 纯编号并列条目，不写标题，不写扩展背景。

## PRD / IPP 中文写作指导

### PRD 指导

- 目标：根据用户的初始需求，输出一份清晰、可执行、适合初级开发者阅读的产品需求文档。
- 默认流程：
  1. 接收用户的功能请求或问题描述。
  2. 先提出 3-5 个最关键的澄清问题，只问会直接影响 PRD 清晰度的点。
  3. 问题必须编号，并为每题提供 A/B/C/D 选项，便于用户用 `1A, 2C` 形式回复。
  4. 在拿到答复后，再生成 PRD；未澄清完成前，不得直接开始实现。
- PRD 推荐结构：
  1. Introduction / Overview
  2. Goals
  3. User Stories
  4. Functional Requirements
  5. Non-Goals
  6. Design Considerations（可选）
  7. Technical Considerations（可选）
  8. Success Metrics
  9. Open Questions
- 若存放到 `/tasks/`，文件名使用 `prd-[feature-name].md`。

### IPP 指导

- 目标：根据用户需求、功能说明或已有文档，生成一份分步骤、可执行的任务清单，指导开发实现。
- 默认流程：
  1. 接收需求。
  2. 先分析范围、功能点、边界与潜在相关文件。
  3. 第一阶段只生成高层父任务，并且默认包含 `0.0 Create feature branch`，除非用户明确要求不要建分支。
  4. 生成父任务后暂停，等待用户回复 `Go`，再进入第二阶段。
  5. 第二阶段补充子任务、Relevant Files、Notes 与完整任务结构。
- IPP 输出要求：
  - 必须包含 `Relevant Files`、`Notes`、`Instructions for Completing Tasks`、`Tasks`。
  - 任务清单必须使用 checkbox。
  - 任务完成后应及时把 `- [ ]` 改成 `- [x]`。
- 若存放到 `/tasks/`，文件名使用 `tasks-[feature-name].md`。

## 实现与记录约束

- 代码实现需与 PRD 和 IPP 一致；若有偏离，必须记录原因。
- 代码实现需与 PRD、系统设计文档和 IPP 一致；若有偏离，必须记录原因。
- 每次更新完成后，可上传到 Git。
- Git 提交说明（commit message / comment）使用中文。
- 功能或文档任务完成后，写入 `docs/progress.txt`，至少包含：时间、完成任务、问题、解决方法。
